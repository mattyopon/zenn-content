## はじめに

「イレブンナインを目指せ」と言われ、InfraSim の可用性計算を小数点 10 桁精度にしてフェイルオーバー 0.1 秒まで高速化した結果、**6.65 Nines** でした。

## 精密計測の結果

```python
# 30-day stress test (v10.4)
Avg Availability = 99.9999777%
Nines = -log10(1 - 99.9999777/100) = 6.65

# つまり年間ダウンタイム = 365 * 24 * 60 * 60 * (1 - 0.999999777)
#                       = 7.03 秒/年
```

| シナリオ | Availability | Nines | 年間 DT |
|---------|-------------|-------|---------|
| 7日 baseline | 100.000000000% | ∞ | 0 |
| 7日 deploys | 99.999996521% | 7.46 | 1.1秒 |
| 30日 stress | 99.999977667% | **6.65** | **7.0秒** |

## なぜ 6.65 で止まるか

### ボトルネック: AZ相関障害

30日シミュレーションで AZ 障害（コンポーネントタイプの33%が同時ダウン）が 1 回発生。このとき:

```
affected = 12 pods × 33% = 4 pods 同時ダウン
failover_time = 0.1s + 0.1s = 0.2s
micro_penalty = 4 × (1/12) × (0.2/300) / 38 × 100 = 0.000006%

× 38 components ≈ 0.00022% total impact → 6.65 nines
```

### イレブンナインに必要な条件

```
11 nines = 99.999999999%
Allowed downtime/year = 0.32 milliseconds

Required failover time: < 0.00001 seconds (10 microseconds)
→ 物理的に不可能（ネットワークの Round Trip > 10μs）
```

## 結論

**6.65 Nines（年間 7 秒のダウンタイム）は、ソフトウェアアーキテクチャで到達可能な実質的な上限**です。

これを超えるには:
- **ハードウェアレベルの冗長化**（FPGA ベースのフェイルオーバー）
- **物理ネットワーク最適化**（InfiniBand、RDMA）
- **OS カーネルレベルの最適化**（リアルタイムカーネル）

これらは「インフラアーキテクチャ」の範疇を超え、**物理制約との戦い**になります。InfraSim はソフトウェアレベルのシミュレーションツールとして、この精度が限界です。
